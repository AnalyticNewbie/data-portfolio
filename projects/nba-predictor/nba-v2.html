v2.0: The Migration

The "Big Rewire." Moving from fragile CSV files to a robust PostgreSQL architecture and upgrading our prediction engine from simple lines to Gradient Boosting trees.

Architecture & Stack

PostgreSQL SQL Views Scikit-Learn Gradient Boosting

We shifted to a classic ELT (Extract, Load, Transform) pattern. Data is ingested raw into PostgreSQL, cleaned via SQL Views, and then fed into a Gradient Boosting Classifier.

schema_migration.sql SQL 
CREATE VIEW v_team_performance AS
SELECT
    team_id,
    game_date,
    pts_scored,
    -- SQL Window Functions replace complex Pandas Loops
    AVG(pts_scored) OVER (
        PARTITION BY team_id 
        ORDER BY game_date 
        ROWS BETWEEN 5 PRECEDING AND 1 PRECEDING
    ) as rolling_avg_pts
FROM raw_games;
model_pipeline_v2.py Python 3.12 
from sklearn.ensemble import GradientBoostingClassifier

# Switching from Linear Regression to Trees

clf = GradientBoostingClassifier(
    n_estimators=100,
    learning_rate=0.1,
    max_depth=3,
    random_state=42

clf.fit(X_train_db, y_train_db)

Lessons & Bottlenecks

What Failed in v1?

Linear Regression assumed that "Points Scored" had a straight-line relationship with winning. It failed to account for context. If the Celtics score 100 vs the Pistons, that's bad. If they score 100 vs the Heat (a slow team), that's good.

The New Bottleneck

While the database is stable, we discovered a data ingestion error. Games played in different timezones (West Coast vs East Coast) are creating "Ghost Games" on Sundays.

Intended Next Steps

We need to fix the Timezone bug discovered during the migration. This will require a refactor of the `date_utils` module.
